# Phase 1.5 — Multi-Language Support (TS/JS) Design Spec

- **Status:** Draft for review
- **Date:** 2026-06-24
- **Builds on:** [Phase 1 memory-service](2026-06-24-agent-dev-workspace-design.md#52-memory-service-the-heart) (merged to `main`).
- **Branch:** `feat/multi-language-parsers`

---

## 1. Problem

The Phase 1 memory service parses **Python only**. Any non-`.py`/`.md` file is skipped by the worker, so it produces zero symbols, edges, or code chunks — invisible to the structure graph and retrieval. To support polyglot projects, the service needs parsers for additional languages.

The parser is the *only* language-specific component: chunking, embedding, resolution, retrieval, MCP, and reconcile all operate on `ParsedSymbol`/`ParsedEdge`/chunks and are reused unchanged.

## 2. Goals / non-goals

**Goals**
- Ingest **TypeScript and JavaScript** (`.ts`, `.tsx`, `.js`, `.jsx`, `.mjs`, `.cjs`) into the existing structure graph + semantic chunks.
- Replace the worker's hardcoded extension handling with an extensible **parser registry**.
- Make edge resolution **language-scoped** so cross-language name collisions don't produce false edges.

**Non-goals (deferred)**
- **Dart** and **SQL**: dropped for now — no official tree-sitter binding meets the "official, mature" bar (Dart binding is a single-maintainer `v0.1.0`; there is no official tree-sitter-org SQL grammar). Revisit when official/mature bindings exist or the "official only" rule is relaxed.
- TS-specific declarations (interfaces, type aliases, enums) — parity with Python's function/method/class model only.
- Project-spec-driven parser configuration — the supported set is a static map (YAGNI).

## 3. Constraints / decisions

| Decision | Value | Rationale |
|---|---|---|
| Languages | TypeScript + JavaScript | Official `tree-sitter` org grammars, current, mature. |
| Symbol scope | function / method / class (parity) | Reuses the existing model + chunker unchanged; tightest first cut. |
| Bindings | `tree-sitter-typescript>=0.23`, `tree-sitter-javascript>=0.25` | Verified official packages: `github.com/tree-sitter/tree-sitter-typescript` (0.23.2), `github.com/tree-sitter/tree-sitter-javascript` (0.25.0). |
| Registry | static extension → parser map | Fixed supported set; no config indirection yet. |
| Resolution | language-family scoped | Prevents a TS `process` resolving to a Python `process`. |
| Schema | no change | `files.language` already exists; reused as the family label. |

## 4. Components

### 4.1 Parser registry (`src/memory/parser/registry.py`)
A static map from file extension to `(language_family: str, parser: LanguageParser)`:

- `.py` → `("python", PythonParser())`
- `.ts` → `("javascript", JsTsParser(grammar="typescript"))`
- `.tsx` → `("javascript", JsTsParser(grammar="tsx"))`
- `.js` / `.jsx` / `.mjs` / `.cjs` → `("javascript", JsTsParser(grammar="javascript"))`

Exposes `code_parser_for(rel_path: str) -> tuple[str, LanguageParser] | None` (returns `None` for unsupported extensions, including `.md`, which keeps using the existing doc path).

**Trade-off:** a static map is the lowest-indirection design; when a language is added, the change is one registry entry + one parser module. Markdown deliberately stays out of the *code* registry (it is doc content, not structured code).

### 4.2 JS/TS parser (`src/memory/parser/js_ts_parser.py`)
One `JsTsParser` class parameterized by grammar (`"typescript"` → `tree_sitter_typescript.language_typescript()`, `"tsx"` → `language_tsx()`, `"javascript"` → `tree_sitter_javascript.language()`). Implements the `LanguageParser` protocol (`parse(path, source) -> ParsedFile`).

Extracts (kinds parity with Python):
- **Symbols:**
  - `function` — `function_declaration`; and arrow/function expressions bound to a `const`/`let`/`var` (name taken from the `variable_declarator`, span covering the declaration so the chunk includes it).
  - `class` — `class_declaration`.
  - `method` — `method_definition` inside a class (qualname `Class.method`).
- **Edges:**
  - `calls` — `call_expression` → callee's last identifier (mirrors the Python parser's callee-name rule).
  - `imports` — `import` statements → module source string; and `require('x')` calls → module name.

Qualname/scope nesting mirrors the Python parser (a scope stack producing `Outer.inner`). Malformed files yield a partial tree; the parser extracts what it can (same tolerance as the Python parser).

**Trade-off:** name-based callee extraction is imprecise (parity with Phase 1's known simplification) — resolution precision is handled by the resolver, not the parser.

### 4.3 Worker refactor (`src/memory/worker.py`)
`process_file` replaces the hardcoded `.py`/`.md` branches with a registry lookup:

1. `code_parser_for(rel_path)` returns `(language, parser)` → parse → `upsert_file_row(rel_path, language, digest)` → `replace_structure` → `resolve_pending_edges` (closure) → `sync_code_chunks`. Returns `"ingested"`.
2. Else if `.md` → existing doc path (`language="markdown"`). Returns `"ingested"`.
3. Else → `"skipped"`.

Hash-skip and missing-file deletion behavior are unchanged.

### 4.4 Language-scoped resolution (`src/memory/repository.py`)
`resolve_pending_edges` currently matches a `calls` edge's `dst_name` against symbols **globally by name**. With multiple languages in one repo this can mint a false cross-language edge (a TS `process()` call resolving to a Python `process`).

Change: scope the match to the **same language family** as the edge's file. Concretely, build the name→symbol-ids index keyed by `(language, name)` (joining `symbols` → `files.language`), and resolve each edge (whose language comes from its own `files.language`) only against same-language symbols. TS and JS share the `"javascript"` family, so they resolve to each other; Python (`"python"`) stays isolated.

`reresolve_all_edges` keeps its shape (reset → resolve). `impact_of` is unchanged — it reads resolved `dst_symbol_id`, which is now correct by construction.

**Trade-off:** resolution is scoped by the stored `language` family label, not by true module-graph reachability — consistent with the name-based v1 model; import-path-precise resolution remains a later escalation.

## 5. Data flow

Unchanged pipeline (`enqueue → worker → reconcile`). Only two steps change: the **parse** step becomes registry-dispatched, and **resolution** becomes language-scoped. Closure, chunk-hash diffing, batched embedding, retrieval, and MCP are untouched.

## 6. Testing

- **JS/TS parser** (`tests/test_js_ts_parser.py`): for TS, TSX, and JS samples — function declarations, arrow-function-in-`const`, classes, methods, call edges, import/`require` edges; correct qualnames.
- **Registry** (`tests/test_registry.py`): each extension → expected `(language, parser type)`; unsupported extension (and `.md`) → `None`.
- **Worker end-to-end** (`tests/test_worker_multilang.py`): ingest a `.ts` file → symbols + `code_chunks` present, `files.language == "javascript"`.
- **Language-scoped resolution** (`tests/test_resolution_multilang.py`): a Python `process` and a TS `process` in the same DB do **not** cross-resolve; a TS file calling a JS function (same family) **does** resolve.

## 7. Dependencies

Add to `pyproject.toml`: `tree-sitter-typescript>=0.23`, `tree-sitter-javascript>=0.25`.

## 8. Out of scope / escalation paths

- **Dart / SQL**: deferred until official/mature bindings exist (or the rule is relaxed for derekstride's community SQL grammar). SQL also needs a *different* symbol/edge model (tables/views/procedures, FK/reference edges).
- **TS type declarations** (interface/type/enum): additive — new kinds + (optionally) extending the chunker to chunk them.
- **Import-path-precise resolution**: replace name-based matching with module-graph resolution.
